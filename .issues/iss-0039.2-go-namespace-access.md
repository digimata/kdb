# ISS-0039.2 :: Go namespace access not mapped to symbol refs

## Summary

`refs -s` finds zero references for Go symbols accessed via package namespace (`pkg.Foo`). This breaks the standard Go import pattern and means Go recall is 0%.

## Root cause

Go imports work differently from other languages — there are no named imports. `import "example.com/proj/pkg"` makes the package available as `pkg`, and symbols are accessed via `pkg.Foo()`.

The current pipeline assumes named imports: `import_names()` in `src/resolve/go.rs:364` returns the package alias (`"pkg"`), not the individual symbol names. The usage scanner finds `pkg` as an identifier, but `insert_usage_row()` looks up `"pkg"` in `symbol_lookup` for the target file, which only contains definition names like `"Foo"`. No match, ref dropped.

The architecture assumes `ResolvedImport.names` contains symbol names that exist in the target file's symbol table. Go violates this assumption because it uses namespace-qualified access.

## Repro

```rust
// tests/refs_eval.rs — go_g01_package_import
eval_refs(
    &[
        ("go.mod", "module example.com/proj\n\ngo 1.21\n"),
        ("pkg/target.go", "package pkg\n\nfunc Foo() {}\n"),
        (
            "cmd/main.go",
            "package main\n\nimport \"example.com/proj/pkg\"\n\nfunc main() {\n\tpkg.Foo()\n}\n",
        ),
    ],
    "pkg/target.go",
    "Foo",
);
// Expected: 1 definition + 1 usage
// Actual:   1 definition + 0 usages
```

## Affected categories

| Test | Category | Description |
|---|---|---|
| G1 | Package import | `import "pkg"; pkg.Foo()` |
| G6 | Type usage | `var x pkg.Foo` |
| G7 | Composite literal | `pkg.Foo{X: 1}` |

## Fix direction

Two possible approaches:

**Option A: Decompose namespace access in the usage scanner.** When scanning Go files, detect `pkg.Foo` as a selector expression where `pkg` matches an imported package name. Extract `Foo` as the usage name and look it up in the target file's symbol table. This keeps `import_names` returning the package alias but adds Go-specific logic to the scanner.

**Option B: Expand `import_names` to include all exported symbols.** When resolving a Go import, enumerate all exported symbols in the target package files and include them in `names`. This makes the existing pipeline work but is more expensive and couples import resolution to symbol extraction.

Option A is cleaner — it matches how Go actually works (namespace access is a language-level concept, not an import concept). Key change site: `src/index/code_refs.rs:345` (`UsageScanner`) — add Go-specific selector expression handling.

## Implemented

- Added Go-qualified usage extraction in `src/index/code_refs.rs` for both `selector_expression` (`pkg.Foo()`) and `qualified_type` (`pkg.Foo`, `pkg.Foo{...}`).
- Extended usage rows to carry optional import binding context so package binding lookup and symbol lookup are separated.
- Updated usage linking to resolve target files by binding name while resolving symbol refs by referenced symbol name.
