# ISS-0039.6 :: Namespace/module qualified access not resolved

## Summary

`refs -s` misses usages accessed through a namespace or module qualifier: `X.name` where `X` is a namespace/module import. The usage scanner sees `X` and `name` as separate identifiers but doesn't connect `name` back to the imported module.

## Root cause

The usage scanner in `code_refs.rs` checks each identifier independently against `ImportedBindings`. For namespace imports like `import * as bar from './bar'`, the binding maps `bar` → `./bar`. When the code says `bar.foo()`, the scanner sees `bar` (matches the binding) and `foo` (no binding). The actual reference to `foo` from `./bar` is never created.

Same pattern in each language:

- **TS**: `import * as bar from './bar'; bar.foo()` — `foo` not resolved
- **Python**: `import foo; foo.bar()` — `bar` not resolved
- **Go**: `import . "pkg"; Foo()` — dot import brings names into scope without qualifier, but we don't expand them

## Affected categories

| Test | Lang | Category | Example |
|---|---|---|---|
| T4 | TS/JS | Namespace import | `import * as bar from './bar'; bar.foo()` |
| P2 | Python | Module import | `import foo; foo.bar()` |
| G3 | Go | Dot import | `import . "pkg"; Foo()` |

## Fix direction

In the usage scanner, when encountering a member expression (`X.name`):

1. Check if `X` maps to a namespace/module import in `ImportedBindings`
2. If so, treat `name` as a reference to a symbol in the resolved file
3. For Go dot imports specifically, all unqualified identifiers that match exported names from the imported package should be treated as references

Key change site: `src/index/code_refs.rs` — `UsageScanner` needs to handle member expressions, not just bare identifiers.
