# ISS-0039.7 :: Wildcard imports not expanded

## Summary

`refs -s` cannot resolve symbols imported via wildcard/glob imports (`use X::*`, `from X import *`). The import is recorded but no individual symbol names are bound, so no usages are found.

## Root cause

Wildcard imports bring all exported names from a module into scope, but the resolver doesn't enumerate those names. `ResolvedImport` has `resolved_path` pointing to the source module and `names = []` (empty). With no names in the binding, the usage scanner has nothing to match against.

Expanding wildcards requires a new capability: "list all exported symbols from this file." This means parsing the target file during import resolution and extracting its public symbol names.

## Affected categories

| Test | Lang | Category | Example |
|---|---|---|---|
| R5 | Rust | Wildcard import | `use crate::foo::*; Bar::new()` |
| P4 | Python | Wildcard import | `from foo import *; bar()` |

## Difficulty

Higher than other recall fixes because it requires:

1. Parsing the target file to extract exported symbols (new capability)
2. Handling `__all__` in Python (controls what `*` imports)
3. Handling visibility rules in Rust (`pub` items only)
4. Potential performance impact — every wildcard import triggers a file parse

## Priority rationale

Low priority because wildcard imports are rare in practice:
- Rust: `clippy::wildcard_imports` lint discourages them
- Python: PEP 8 and most linters flag `import *`
- TS/JS: no wildcard import syntax (namespace imports are tracked separately in 0039.6)
- Go: no equivalent (dot imports are tracked in 0039.6)
