# ISS-0039.5 :: Re-exports not followed

## Summary

`refs -s` cannot resolve symbols that are accessed through re-export intermediaries. This is the single biggest architectural gap — re-exports are idiomatic in every supported language and account for a large fraction of real-world imports.

## Root cause

Import resolution stops at the first file it finds. When that file re-exports the symbol from elsewhere, we record the intermediary as the definition site. The actual definition lives in a different file that we never reach.

Example (TS barrel file):

```
components/index.ts:   export { Button } from './Button';
components/Button.tsx: export function Button() { ... }
app.tsx:               import { Button } from './components';
```

Resolution: `'./components'` → `components/index.ts`. We record `Button` as defined in `index.ts`. But `index.ts` only re-exports it — the real definition is in `Button.tsx`. The symbol lookup in `index.ts` finds no `Button` definition, so the reference is dropped.

Same pattern in every language:
- **Rust**: `pub use inner::Foo;` in `mod.rs`
- **Python**: `from .core import Engine` in `__init__.py`
- **TS/JS**: `export { Foo } from './foo'` in `index.ts`
- **Go**: N/A (Go has no re-export mechanism)

## Affected categories

| Test | Lang | Category | Example |
|---|---|---|---|
| R4 | Rust | `pub use` re-export | `mod.rs` does `pub use inner::Foo;` |
| T5 | TS/JS | Barrel re-export | `index.ts` does `export { foo } from './inner'` |
| T6 | TS/JS | Default re-export | `export { default as foo } from './inner'` |
| P5 | Python | `__init__.py` re-export | `__init__.py` does `from .core import Engine` |

## Fix direction

Add a **re-export following pass** after initial import resolution, during index build. For each `ResolvedImport` where `symbol_lookup` fails to find the symbol in the resolved file:

1. Parse the resolved file for re-export statements
2. If the symbol appears in a re-export, resolve the re-export's target
3. Update the `ResolvedImport.resolved_path` to the final definition site
4. Limit to one hop (covers 95%+ of real-world cases; deep chains are rare)

This is a new phase in the index pipeline (`src/index/code_refs.rs`), not a per-resolver change. Each resolver may need a small helper to extract re-export statements from a file, but the orchestration is language-agnostic.

Key change sites:
- `src/index/code_refs.rs` — add re-export following pass after `ImportedBindings` construction
- `src/resolve/tsjs.rs` — helper to extract `export { X } from './Y'` patterns from a file
- `src/resolve/rust.rs` — helper to extract `pub use X::Y` patterns from a file
- `src/resolve/python.rs` — helper to extract `from .X import Y` patterns from `__init__.py`

### Open questions

- Should we follow more than one hop? Rust crate facades can chain `pub use` through multiple `mod.rs` files.
- Should re-export extraction reuse the existing resolver infrastructure or be a lightweight dedicated parser?
- How do we handle `export *` / `pub use crate::*` re-exports? (Probably defer — same problem as wildcards.)
