# ISS-0039.14 :: TS member access on imported value not resolved

## Summary

`import { ToastState } from './state'; ToastState.subscribe()` — `ToastState` is not found as a usage when accessed via member expression. This is distinct from namespace imports (0039.6 / T4) — here `ToastState` is a named import used as a value, not a module.

## Root cause

The usage scanner finds `ToastState` as an identifier, but when it appears as the object in a member expression (`ToastState.subscribe()`), the scanner may be filtering it out or the tree-sitter node kind differs from a standalone identifier.

This is different from the namespace access pattern (0039.6) where a *module* is imported and properties are accessed. Here the import is a regular named import — the identifier `ToastState` itself should be counted as a usage regardless of what comes after the dot.

## Evidence

Tested against sonner: `ToastState` imported from `./state`, used as `ToastState.subscribe()` and `ToastState.dismiss()` — not counted as references.

## Fix direction

The member expression `ToastState.subscribe()` has `ToastState` as the `object` child. The scanner should still recognize `ToastState` as a usage identifier even when it's the object of a member access.

Key change site: `src/index/code_refs.rs` — check if identifiers in `member_expression.object` position are being excluded by the parent-node filter.
