# ISS-0039.8 :: tsconfig path aliases not resolved

## Summary

`refs -s` misses imports that use tsconfig `paths` aliases (e.g. `@/foo` → `src/foo`). These are extremely common in real-world TS projects — n8n, Next.js apps, and most modern TS codebases use them.

## Root cause

We create one global `oxc_resolver::Resolver` with `TsconfigDiscovery::Auto`, which tries to auto-discover tsconfig from CWD. In monorepos, each package has its own `tsconfig.json` with its own `paths` — the global resolver doesn't find them.

## Repro

n8n monorepo: `packages/cli/tsconfig.json` has `"paths": { "@/*": ["./*"] }`.

```
// packages/cli/src/commands/start.ts
import { Server } from '@/server';  // ← not resolved (should → packages/cli/src/server.ts)
```

`kdb refs packages/cli/src/server.ts -s Server` returns 1 (definition only).

## Fix: per-package resolvers

Turborepo solves exactly this problem. The pattern (from `crates/turbo-trace/src/tracer.rs`):

1. Create **one resolver per package**, passing that package's `tsconfig.json` as an explicit `config_file`:

```rust
options.tsconfig = Some(TsconfigOptions {
    config_file: package_tsconfig_path.into(),
    references: TsconfigReferences::Auto,  // handles extends chains
});
```

2. For files with a closer tsconfig (e.g. nested `src/tsconfig.json`), clone the resolver with updated options.

3. Use `TsconfigReferences::Auto` (not `TsconfigDiscovery::Auto`) so the library handles `extends` chains automatically.

### Changes needed

1. **Package boundary detection**: during `WorkspacePackages` build, record which packages have a `tsconfig.json`. Package boundaries come from `package.json` `workspaces` field (already parsed).

2. **Per-package resolver construction**: instead of one `Resolver` on `TsjsResolver`, hold a map of `package_root → Resolver`. For each source file, look up its containing package and use that resolver.

3. **Fallback**: for files not inside any package (monorepo root scripts, etc.), fall back to a root-level resolver with the root `tsconfig.json` if one exists.

### Key change site

`src/resolve/tsjs.rs` — `TsjsResolver::new()` currently creates one `Resolver`. Change to create one per package with its tsconfig.

### Dependency migration

Migrate `oxc_resolver` 11.18.0 → `unrs_resolver` 1.11.1 (same project, renamed). Single import site: `src/resolve/tsjs.rs`. Bundle with this fix since we're switching from `TsconfigDiscovery` to `TsconfigOptions` / `TsconfigReferences` anyway.
