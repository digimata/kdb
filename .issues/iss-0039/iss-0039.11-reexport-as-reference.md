# ISS-0039.11 :: Re-export statements not counted as references

## Summary

`export { foo } from './bar'` re-exports `foo` but the re-export itself is not counted as a reference to `foo`. This means barrel files that re-export a symbol don't show up in `refs -s` results.

## Rationale

A re-export is semantically a reference — the barrel file depends on and forwards the symbol. An agent looking at `refs -s` to understand impact should see the re-export as a usage site.

## Evidence

Tested against sonner: `export { toast … }` not counted as reference.
Tested against shadcn-ui: `export { searchRegistries }` not counted.

## Affected languages

- **TS/JS**: `export { foo } from './bar'`, `export { default as Foo } from './bar'`
- **Rust**: `pub use inner::Foo;`
- **Python**: `from .inner import Foo` in `__init__.py` (when it's a re-export, not local use)

## Fix direction

The re-export following pass (phase 2 of iss-0041) already parses re-export statements. When it encounters one, it should also emit a reference from the re-exporting file to the original definition.

Key change: during re-export extraction, create a `SymbolRef` for the re-exporting file in addition to following the chain.
