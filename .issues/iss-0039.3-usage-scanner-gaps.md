# ISS-0039.3 :: Usage scanner misses parameter types and JSX identifiers

## Summary

Two gaps in `UsageScanner` / `is_declaration_identifier` cause `refs -s` to miss valid usages:

1. **Rust R9**: Type identifiers in function parameter positions are filtered out
2. **TS/JS T11**: JSX component references use `jsx_identifier` node kind, not matched

Both are in `src/index/code_refs.rs`.

## Gap 1: Parameter type filter (R9)

### Root cause

`is_declaration_identifier()` at `code_refs.rs:436` checks if the parent node kind is `parameter`. This is intended to filter out parameter *names* (e.g. `_x` in `fn f(_x: Bar)`), but it also filters out type identifiers in the same parent scope. The `Bar` type_identifier has parent `parameter`, so it gets classified as a declaration and skipped.

In contrast, `Vec<Bar>` (R10) works because `Bar`'s parent is `type_arguments`, not `parameter`.

### Repro

```rust
// tests/refs_eval.rs â€” rust_r09_type_in_signature
eval_refs(
    &[
        ("src/lib.rs", "pub mod target;\npub mod caller;\n"),
        ("src/target.rs", "pub struct Bar;\n"),
        ("src/caller.rs", "use crate::target::Bar;\npub fn f(_x: Bar) {}\n"),
    ],
    "src/target.rs",
    "Bar",
);
// Expected: 1 definition + 1 usage
// Actual:   1 definition + 0 usages
```

### Fix direction

In `is_declaration_identifier`, the `parameter` parent check needs to be more precise. Instead of matching on parent kind alone, check whether the node is the `name`/`pattern` field of the parameter (the parameter name), not the `type` field. A `type_identifier` child of `parameter` is a usage, not a declaration.

Alternatively, check the node kind: if the node is a `type_identifier`, it should never be treated as a declaration identifier regardless of parent.

## Gap 2: JSX identifier (T11)

### Root cause

`is_usage_identifier()` at `code_refs.rs:398` matches `identifier` and `type_identifier` for TSX files, but JSX component names in `<Foo />` use the `jsx_identifier` node kind in tree-sitter's TSX grammar.

### Repro

```rust
// tests/refs_eval.rs â€” tsjs_t11_jsx_component
eval_refs(
    &[
        ("src/target.tsx", "export function Foo() { return null; }\n"),
        ("src/caller.tsx", "import { Foo } from './target';\nexport default function App() { return <Foo />; }\n"),
    ],
    "src/target.tsx",
    "Foo",
);
// Expected: 1 definition + 1 usage
// Actual:   1 definition + 0 usages
```

### Fix direction

Add `"jsx_identifier"` to the match arm for `CodeLanguage::Tsx` in `is_usage_identifier()`. May also need to verify that `is_import_identifier` doesn't false-positive on JSX identifiers.
